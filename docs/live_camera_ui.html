Camera operations

Live camera UI

Run this on the main PC to verify multi-view camera liveness, framing, frame IDs, and daemon health before recording or robot motion.

Start the monitor

# Capture-PC camera daemons must already be up.
python src/camera/reset_cameras.py

# Start the browser monitor on the main PC.
python src/capture/camera/live_monitor.py \
  --host 0.0.0.0 \
  --port 8792

Open http://<main-pc>:8792, then press Start stream.

Runtime path

  • SSH-launches src/capture/camera/stream_client.py on capture PCs.
  • Creates remote_camera_controller("camera_live_monitor").
  • Runs rcc.arm(syncMode=False, fps=15) by default.
  • Turns on the stream sink with rcc.set_stream(True).
  • Collects JPEG previews through DataCollector on port 1234.

Scope

Preview, not dataset recording

stream_client.py downsamples each camera image by 8x and publishes per-camera JPEG frames. The browser renders one tile per camera instead of a server-side merged image, so all 20 cameras remain visible and slow frames do not block the whole view. Use this UI for framing, liveness, frame ID progress, and daemon health. Use capture scripts for saved datasets.

Control
Action

Start stream
Launch preview clients, arm acquisition, enable stream sink.
Per-camera preview tiles appear in the browser.
Stop
Stop acquisition, close collector, send exit to stream clients.
Remote preview clients exit instead of staying alive on capture PCs.
Refresh
Reload status JSON.
Capture-PC status, frame IDs, and source age update.

Troubleshooting

If the UI shows no frames

Check daemons

  • Each capture PC should run src/camera/server_daemon.py.
  • Use python src/validate/camera_system/remote_camera_controller.py for daemon health and camera counts.

Check transport

  • Use python src/validate/data_sender/main.py for data return.
  • Use --remote-log to write stream-client output to test.log on capture PCs.

  --port 8792 \
  --camera-fps 15 \

  --pc capture1 \
  --pc capture2

python src/capture/camera/live_monitor.py --no-launch-clients
